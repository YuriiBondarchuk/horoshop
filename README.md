## Ключові розбіжності

**1. Пароль зберігається у відкритому вигляді і повертається у відповіді**

- Згідно з отриманим ТЗ, пароль обмежений до 8 символів і повертається у відповіді при створенні користувача. У реалізації це вимога виконана буквально.
- Також ця особливість не дає можливості хешувати пароль і зберігати його в прихованому вигляді. Це теж реалізовано виключно для реалізації ТЗ.

**2. Тести**
- Написані інтеграційні тести демонструють працездатність основних сценаріїв логіну.
- Тести не покривають всі case а написані тільки щоб показати їх працездатність та можливість до розширення.

## Причина такої реалізації

**Реалізація виконана відповідно ТЗ, яке вимагало певної поведінки (обмеження пароля, повернення пароля у відповіді).** 


## Як відтворити поточну поведінку

- **Логін:** `POST /v1/api/login` з JSON `{ "login": "...", "pass": "..." }` повертає JWT.
- **Все решта доступне за роутом:** `POST /v1/api/users` та повертає у відповіді поля згідно з ТЗ.
